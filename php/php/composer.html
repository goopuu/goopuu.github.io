提示

本章不是学习完整 Composer 的章节，仅仅学习使用 Composer 安装依赖，以及实现自动加载。

安装依赖：
在目录下手动创建 composer.json 文件，然后在里面声明依赖，最终执行安装。
或者直接通过命令行安装依赖，会自动生成 composer.json文件。

# 手动创建 composer.json
{
    "require": {
        "xxx/xxx": "1.*"
    }
}

# 然后执行命令安装依赖
composer install

# 直接通过命令安装
composer install xxx/xxx 1.*

如上操作，会安装指定的依赖：

  • 如果存在 composer.lock，其实安装操作是根据这个文件安装的
  • 如果不存在 composer.lock，才会根据 composer.json 中的声明安装依赖，并且最终会生成一个 composer.lock

更新依赖：
此时一定是根据 composer.json 文件来更新的了，比如上述的依赖包当时安装的版本为 1.1，现在这个包已经被作者更新到了 1.2，甚至 2.2，如此如果更新依赖的话，会根据声明 1.* 更新为 1版本中最新的那个版本，即1.2，而不会更新到 2.2。

# 全部更新
composer update

# 指定更新具体包
composer update xxx/xxx

# 指定更新具体包为新的版本
composer update xxx/xxx 2.*      // 这个时候就会更新到 2.2 了

版本管理：
原则上说，composer.json 和 composer.lock 都应该提交到版本控制系统中，如此各个开发的依赖包的版本才会一致。

自动加载：
默认安装的依赖，会在安装目录下生成一个 vendor 的目录，所有安装的依赖都在里面。
只需要引入 vendor/autoload.php，就能够使用 Composer 的自动加载功能，并且安装的所有依赖都已经实现了自动加载。

# 入口文件
require_once 'vendor/autoload.php';

// 此时就可以实现自动加载安装的依赖了
use xxx/xxx;
$xxx = new xxx();

还有一种，实现自己编写的类库的自动加载，比如控制器、模型、类库等等，只需要在 composer.json 中声明："autoload": {“psr-4": {命名空间: 目录路径}}

{
    "autoload": {
        "psr-4": {
            "App\\": "app/",
            "Database\\": "database/factory/",
            "Lib\\": "shared/library/"
        }
    }
}

注意

注意，目录路径的相对路径，是相对于 composer.json 所在目录来说的。
注意，编写了、修改了 autoload，需要重新执行 composer install 生成新的 autoload.php才能实现更新自动加载。
