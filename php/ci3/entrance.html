# 1. 应用程序入口

nginx 需要一个入口让 linux 通过它进行管理，php-fpm 需要一个入口与 nginx 进行交互。
同理，应用程序也需要一个入口，让 nginx 通过它与应用程序交互。

# 1.1 index.php

下载框架包后，在根目录有一个 index.php，它是“默认的”入口文件。

提示

入口文件可以任意放置，但是要注意一些问题：

  • 入口文件所在目录应该是安全访问的（其他重要文件不能直接被访问）
  • nginx 中需要配置正确的入口文件路径
  • 入口文件位置修改，则框架核心的部分源代码也可能需要修改

# 2. 入口中的操作

# 2.1 设置环境

define('ENVIRONMENT', isset($_SERVER['CI_ENV']) ? $_SERVER['CI_ENV'] : 'development');

注意

它是通过服务器（nginx）设置的环境变量（服务器环境变量）CI_ENV来判断当前环境的。

如果服务器环境变量 CI_ENV 为设置，则为 development开发环境，否则为设置的变量值作为环境。

那么这样的处理就有几个问题：

  • 如果是 cli 项目，它是不需要 web服务器 的，这个变量设置与否都没有意义，那么默认情况下将永远是 development开发环境
  • 如果是开发、运维严格区分的公司，这种应用程序的设置需要运维人员来处理（我之前待过一家公司，开发人员永远不允许直接、间接运维，实在需要，必须提工单层层同意后，运维人员才会处理）

# 2.2 环境限制

如果服务器上的环境变量设置类似：$_SERVER['CI_ENV'] = 'aaa' 这样的值，显然应用程序不会支持的。
框架默认支持三种环境：

  • development
  • testing
  • production 其他任何环境变量值都视作错误：
header('HTTP/1.1 503 Service Unavailabel.', TRUE, 503);
echo 'The application environment is not set correctly.';
exit(1);

所以，如果要增加环境，需要修改这里的源代码。如果这个环境做成配置的化，修改配置比直接修改源代码更好。

# 2.3 环境限制错误处理

上述环境限制了只支持三种环境，针对每一种环境，关于错误处理有不同的限制：

  • development：展示错误信息，报告所有错误
  • testing 和 production：不展示错误信息，仅报告致命错误

注意

展示错误信息：表示是否在客户端（web、cli等）显示发生错误的信息。
报告错误级别：表示在支持的错误级别中，哪些错误可以被“捕获”。

关于错误处理和捕获以及异常，在 PHP 中会有专门章节学习。

# 3. 全局目录常量

源码中，关于全局目录常量的定义，都是相对于当前目录来说定义的，因此上述也说过，入口文件如果自定义位置，这里设置的全局目录常量的定义也需要调整。

# 3.1 系统目录 system

常量：BASEPATH，指代 system 目录，路径为：xxx/system/。

# 3.2 应用目录 application

常量：APPPATH，指代 application 目录，路径为：xxx/application/。

如果自定义了应用目录结构，这些值都需要修改设定。
