# 1. 了解 git 如何保存信息

在《认识 Git》一节中介绍了，git是 直接存储快照，即每一次提交都保存了当前项目的完整快照（之前也说了，不是真的完整的快照，那些未修改的仅仅做了一个指针引用之前的版本即可）。接下来稍微详细的理解 git 是如何保存信息的。

# 1.1 四个对象

对象名称 说明
快照对象（blob对象） 就是指文件的快照（有修改的则为新的文件快照对象，没有修改的则为上一次提交的文件快照对象）
树对象 它记录当前提交的所有快照对象的检验和（哈希值）和目录结构
提交对象 它包含了指向当前提交的树对象的指针，以及其他信息
父对象 当前提交对象的另一个指向对象，首次提交没有父对象，可以理解为上一次的提交对象

上一节说到，每个提交对象（除了首次提交）都指向另一个对象：父对象。

# 2. 分支

分支的本质仅仅是指向提交对象的可变指针。
每个仓库，都有一个默认的分支（master 或 main）。
它包含两个关键点：

  • 指向提交对象
  • 可变指针 也就是说，分支总是指向提交对象的，并且它很有可能随时在变化，上一刻指向一个提交对象，下一刻又指向另一个提交对象。
    那么，分支的可变情况到底是如何进行的呢？
    原则就是：分支指向当前最新的提交对象。即每次提交，产生提交对象，那么分支则指向当前提交对象，以此类推的话，它会跟着提交对象向前移动。

# 2.1 创建分支

注意

创建分支，是基于“当前提交对象”来创建的。
通常我们会说创建分支是在某某分支上切出一个分支，是不严谨的。

git branch branchname
git branch testing

注意

创建分支，仅仅在当前提交对象上创建了一个新的分支，但是工作分支并没有切换到新的分支上，除非手动切换。

# 2.2 HEAD 指针

如上如果创建了一个新的分支，现在至少就存在两个分支：master 和 testing。那么 git 是如何知道当前是在哪个分支上工作呢？
git 内部还又一个特殊的指针，它指向了当前工作分支，名为 HEAD。 如果仅仅是创建分支，那么如下图所示，工作分支还是在 master 上：

要将工作分支切换到新的分支，即需要将 HEAD 指针指向新的分支，需要切换分支。

# 2.3 切换分支

git checkout branchname
git checkout testing

# 或者在创建分支的时候同时切换到新的分支
git checkout -b testing

此时，工作分支才真的为新的分支，HEAD 才指向新的分支：

# 2.4 基于分支的提交

所有的提交，都是基于当前工作分支的，那么当存在多个分支时，就存在一个场景：某些分支的版本停留在它提交的时候；而切出的新的分支一直在提交，版本比其他分支更新： 图示中可以看到，master 分支保存的数据还是在上一个提交版本，而新的分支已经更新了版本。
再回过头来理解上面介绍的 git 保存信息。每个提交对象指向了当时提交的树对象，树对象指向了当时提交的文件快照。
那么图示中对于 master 分支来说，它此时保存的信息就是那个提交对象时的数据，比 testing 当前提交对象保存的信息要落后版本，可以通过再次切换分支到 master 上查看，会发现新提交的数据看不到了。

# 2.5 分支分叉

现在假设基于上图的情况，切换到了 master 分支，并且在当前分支（master）上再做修改，就会发现一个特别的情况，在某个提交对象上，master 和 testing 分支基于那个提交对象产生了分叉： 其实可以同理理解，就算不是在同一个提交对象上进行不同分支的提交，总之会产生分叉分支的。
这个时候就有问题了，如果既需要 master 分支上最新的提交，又需要 testing 分支上最新的提交；或者说（针对上上一个图，即刚切换到 master 分支，还未提交新的版本前）切换回旧分支时也需要新分支上的提交，此时该怎么办？那就需要分支合并了。

# 2.6 分支合并

同样以上述图示的形式来理解分支合并。
假设这样的场景：
当前在 master 分支上。
然后基于最新的提交对象切出了一个新的分支 testing，在 testing 分支上不断的提交前进。
现在突然来了一个需求，要求立马修复一个 bug，那么，此时 testing 上的工作如果完成了，那么可以切换到 master 分支，然后合并 testing 上的所有修改，然后切出新的分支修复 bug，甚至都不用合并再切新的分支，只需要继续在 testing 分支上修复 bug 就行（但是这种做法对于测试、发布流程不严格要求的情况才行，否则针对每一个版本都需要严格测试、严格发布的还是必须切新的分支）。
但如果 testing 分支上的开发并没有完成，还不能上线，那么就必须要暂时停止 testing 上的工作然后切出新的分支立马修复 bug。此时应该将 testing 分支上的已修改、已暂存的数据保存，有两种做法，一种直接提交，一种 stash 缓存（后续学习），这是因为在切换分支时，如果不保持当前分支的干净状态（还有已修改、已暂存）那么切换可能不成功，就算切换成功了可能导致不应该出现了修改出现在了新切出的分支上。
此时如果保持了 testing 分支上没有已修改、已暂存的数据，那么切换到 master 分支，然后再基于 master 分支的最新提交对象切出一个新的分支 fix 分支，然后在该分支上不断提交。

这个场景如果以图示来看的话，类似如下：

接下来就是分支合并了：

git merge branchname

# 假设修复 bug 的分支已经完成提交
git checkout master
git merge fix
git branch -d fix
git commit -a -m '修复 bug'
git push

提示

一般来说，如果时临时分支，合并后应该删除。

# 2.7 删除分支

# 要删除的分支上没有已修改、已暂存，切换到其它分支上后删除
git checkout master
git branch -d branchname

# 要删除的分支上有已修改、已暂存
# 要么先切换到该分支，然后提交，然后切换到其他分支上进行删除
git checkout needDelete
git commit -a -m 'xxx'
git checkout master
git branch -d needDelete
# 要么直接强制删除，即放弃那个分支上的所有修改
git checkout master
git branch -D needDelete
