除了本地仓库，其他仓库都可以称之为远程仓库，甚至本地磁盘的其他非当前仓库也可以称为远程仓库。

# 1. 添加远程仓库

# 1.1 克隆，自动添加

通过克隆其他仓库在本地形成本地仓库，会自动将克隆的仓库作为本地仓库的远程仓库，并自动命名为 origin。

git clone xxx

# 1.2 手动添加远程仓库

git remote add remotename xxx

提示

可以对本地仓库添加任意多个远程仓库。
远程仓库，可以理解为当前本地仓库的源头，可以有多个源头。

# 2. 本地添加的远程仓库重命名、删除

# 重命名
git remote rename oldname newname

# 删除
git remote remove remotename

# 3. 理解远程分支和本地分支的关系

远程仓库上的分支和本地仓库上的分支，原则上是没有任何关联的，就算它们的名称一模一样。 要使远程分支与本地分支有关联，可以在本地对远程分支进行跟踪，但它是非必要的。

# 3.1 远程跟踪分支

远程跟踪分支存在于本地。
它是本地对远程仓库对应分支的一个引用。
可以把本地的远程跟踪分支理解为远程分支的别名，它就代表了远程分支的信息。
一般来说，远程跟踪分支命名为 remotename/branchname，比如 origin/master，origin/develop等。
克隆，自动创建本地远程跟踪分支 origin/mster 或 origin/main。
还可以手动创建远程跟踪分支。

# 3.2 跟踪分支

跟踪分支，是从远程跟踪分支上切出的一个本地分支，并且与远程分支“直接关联”。
比如克隆时，会自动将远程仓库在本地仓库上命名为 origin，同时自动创建本地的远程跟踪分支 origin/master，同时自动创建本地的跟踪分支 master，本地的 master 分支此时与远程的 master 分支自动直接关联，因此可以使用类似 git pull 或 git push 的操作。

# 3.2.1 手动创建跟踪分支

# 假设本地存在一个 远程跟踪分支： origin/testing
git chekcout -b testing origin/testing
# 如果不需要重命名本地跟踪分支的话，可以如下简写
git checkout --track origin/testing

注意

跟踪分支，一定是根据远程跟踪分支来的，必须在远程跟踪分支的基础上切出的新分支。

# 假设本地根本不存在一个 远程跟踪分支：origin/fix
# 要么是远程没有该分支（fix)；要么是远程有该分支，但是本地还没有拉取并在本地生成 远程跟踪分支(origin/fix)
# 执行如下操作并不成功
git checkout -b fix origin/fix
# 此时会报错，大致：fatal: 'origin/fix' is not a commit and a branch 'fix' cannot be created from it

# 3.2.2 手动推送跟踪分支

上面手动创建跟踪分支，是远程上已经存在某个分支，然后本地拉取后基于远程的那个分支在本地创建的远程跟踪分支来创建本地跟踪分支。
但是更常见的场景是，本地创建了一个分支，经过不断的提交后，最后决定希望将这个分支作为长久开发分支，并推送到远程；或者这个分支上的开发还未完成，但是需要在不同的地方继续开发（比如公司和家里），那么需要将它推送到远程，然后在回家的时候拉取这个分支的数据继续开发，那么此时就需要将该分支进行跟踪：

git push --set-upstream origin xxx
git push --set-upstream origin lts

此时，它会反向操作实现跟踪分支，即它会在远程创建分支 lts，同时在本地创建远程跟踪分支 origin/lts，同时将本地的分支 lts 作为跟踪分支与远程的 lts 直接关联。

# 3.3 取消跟踪

可以直接删除本地的跟踪分支。
或者如果想保留本地的分支，但是不需要跟踪，则取消来源即可：

git checkout develop
git branch --unset-upstream

# 3.4 删除远程跟踪分支

有的时候会发现，本地跟踪分支已经删除，远程分支也删除了，但是本地还存在远程跟踪分支，则手动删除：

git branch -d -r 远程跟踪分支
git branch -d -f origin/develop

# 4. 拉取远程仓库上的版本，合并到本地仓库

# 4.1 fetch 仅拉取，然后需要手动合并

# 拉取远程仓库上的所有数据 包括数据、分支等
git fetch origin

# 仅拉取远程仓库的指定分支
git fetch origin/testing

注意

git fetch，仅做了拉取工作，即把远程仓库上的数据拉取到本地数据库中，但是光拉取，在本地是不能体现远程数据的，需要手动合并。
git fetch，会拉取远程仓库上的数据和分支情况，并在本地创建对应的 远程跟踪分支（就算是拉取具体的远程分支，同理也会在本地创建对应分支的 远程跟踪分支）。

# 合并具体分支
git merge origin/develop

此时，远程仓库上的 develop 分支上的数据才会合并到当前分支。

# 4.2 pull 拉取并自动合并

如果本地已存在跟踪分支，那么可以更加简单的拉取操作。

# 切到跟踪分支上
git checkout develop

# 直接拉取，并自动合并
git pull

当然，就算是跟踪分支，也可以使用手动的方式拉取，然后手动合并。

# 5. 推送本地仓库的版本到远程仓库

# 5.1 推送到指定分支

git push origin develop

# 5.2 推送到跟踪分支

如果是跟踪分支，可以更简单：

git push

当然，就算是跟踪分支，也可以指定分支来推送。

# 6. 删除远程分支

删除本地分支上一节已经说过了，但是如果是跟踪分支的话，删除了本地，远程上还是存在的，很多时候可能希望同时删除本地、以及远程上的对应分支。

git push origin --delete develop

此时一般要保证被删除的本地分支上为干净状态（没有已修改、没有已暂存），否则可能失败。
